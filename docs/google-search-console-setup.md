# Google Search Console Setup — KickOracle

Goal: get `kickoracle.com` indexed in Google fast. The site currently has 2 indexed pages and zero organic traffic. Follow these steps in order.

## 1. Verify domain ownership (use DNS)

Recommended: **Domain property** verification via DNS TXT record. This covers `kickoracle.com` plus every subdomain and every protocol (`http`, `https`, `www`, `m.`) in a single property.

1. Open <https://search.google.com/search-console> and sign in with the Google account that should own search data.
2. Click **Add property** → choose **Domain** (not URL prefix).
3. Enter `kickoracle.com`.
4. Google shows a TXT record like `google-site-verification=<token>`.
5. In your DNS provider (Vercel/Cloudflare/registrar), add a TXT record:
   - **Name/Host**: `@` (apex)
   - **Type**: `TXT`
   - **Value**: the full `google-site-verification=...` string Google shows
   - **TTL**: 3600 (or default)
6. Wait for DNS propagation (usually < 5 min; up to 24h). Click **Verify** in GSC.

Avoid the HTML file method — it only verifies one origin and breaks if you ever move hosts.

If you also want the legacy URL-prefix property (useful for some reports), repeat with property type **URL prefix** = `https://kickoracle.com`. Domain verification covers verification for URL-prefix sub-properties.

## 2. Submit the sitemap

Once verified:

1. In GSC, select the `kickoracle.com` property.
2. Sidebar → **Sitemaps**.
3. Enter `sitemap.xml` (the field auto-prefixes the domain) and click **Submit**.
   - Full URL: <https://kickoracle.com/sitemap.xml>
4. Status should flip to **Success** within a few minutes. Google will report URLs discovered (expect a few thousand).

The sitemap is generated by `src/app/sitemap.ts` and rebuilt on every deploy. It includes hub pages, all 48 team pages, all host city pages, every player profile, head-to-head compare pages, and recent blog posts — each with `hreflang` alternates for all 19 supported locales.

Robots.txt already advertises this sitemap, so Google would eventually find it — submission just accelerates discovery.

### Rate limits

- Manual **Request Indexing** is limited to roughly 10-12 URLs per day per property. Hitting the cap returns a quota error — do not retry, just resume tomorrow.
- Each request takes 1-2 minutes for the live test before the button becomes clickable.
- Do not bulk-request the same URL more than once a week — extra requests do not speed up indexing.
- For programmatic submission at scale (entire team rosters), use the Indexing API only for `JobPosting` and `BroadcastEvent` schema types. Player pages do **not** qualify; manual Request Indexing or natural discovery via sitemap + internal links is the only sanctioned path.

## 4. Pages report — what to look for

After 3-7 days, open **Indexing** → **Pages**.

You will see buckets:

- **Indexed**: target this to grow from ~2 to several hundred within 2 weeks. Click through and confirm the actual canonical English URLs are listed (not just `/de/` and `/tr/` paths).
- **Not indexed**: drills into reasons. Common reasons you may see and what to do:

| Reason | What it means | Action |
| --- | --- | --- |
| **Discovered – currently not indexed** | Google found the URL but did not crawl it. Crawl budget / low-quality signals. | Add more internal links from indexed pages, build at least one external backlink, ensure page renders fast (LCP < 2.5s). Wait 7-14 days, then Request Indexing on the most important offenders. |
| **Crawled – currently not indexed** | Google crawled but decided not to index. Content quality or duplication concern. | Check the page is genuinely useful (not a near-empty squad shell). Add unique copy, FAQs, or stats. Make sure canonical tag points to itself, not another locale. |
| **Page with redirect** | URL redirects; not the canonical. | Usually fine — confirm the redirect target ends up indexed. |
| **Alternate page with proper canonical tag** | A non-canonical locale was discovered. | Normal for the 18 non-English variants. They show up via hreflang once the English page is indexed. |
| **Duplicate without user-selected canonical** | Google picked a different URL as canonical. | Audit `alternates.canonical` in the page's metadata. Ensure self-canonical for each locale. |
| **Soft 404** | Page returned 200 but looks empty. | Check the dynamic route renders real content. Often caused by missing data fetches. |
| **Blocked by robots.txt** | Path is disallowed. | Should only affect `/api/` and `/auth/`. Anything else here is a bug. |

Filter by **Source: Submitted in sitemap** to focus on URLs we are actively pushing.

## 5. International targeting — hreflang

GSC no longer has a dedicated "International Targeting" report in the new UI, but hreflang is enforced via the sitemap's `<xhtml:link rel="alternate" hreflang=...>` tags and the `<link rel="alternate">` tags in each page's `<head>`.

Validation checklist:

1. In GSC, open any indexed URL via **URL Inspection** → **View crawled page** → **HTML**. Search for `hreflang`. You should see 19 entries plus `x-default`.
2. Use the third-party tool <https://hreflang.org/> or <https://technicalseo.com/tools/hreflang/> to validate the sitemap against the page tags.
3. If Google reports **"No return tags"** errors, it means locale A links to locale B but B does not link back. Confirm `src/app/[locale]/layout.tsx` produces matching alternates for every locale.
4. Re-submit the sitemap after any hreflang change.
5. Run `npm run verify:seo` to programmatically check the live site against this checklist (robots.txt, sitemap.xml, canonical/hreflang/JSON-LD on representative pages). Set `BASE_URL` to point at staging.

The site currently has 2 indexed pages, both non-English (`/de/`, `/tr/`). That's a hreflang misconfiguration symptom — Google indexed the variants without the English canonical. Once `/en/` versions are submitted and crawled, they should be promoted to canonical and the German/Turkish pages will appear in their respective locale SERPs.

## 6. Ongoing monitoring (weekly cadence)

- **Performance** report: track impressions and clicks per query. Filter by country and device. Expected first signals: brand searches like `kickoracle` within 2-3 days of indexing.
- **Indexing → Pages**: re-check weekly. Investigate every new "Not indexed" reason.
- **Experience → Core Web Vitals**: ensure mobile LCP < 2.5s, INP < 200ms, CLS < 0.1.
- **Manual Actions**: should be empty. If anything appears, treat as P0.
- **Security Issues**: same.

If after 14 days the indexed count is still < 50, the issue is one of:
- Bots are being blocked by middleware (check `src/middleware.ts`).
- Pages are returning soft 404 due to missing data at build time.
- Canonical tags are pointing to the wrong locale.
- Site has thin content / lacks external backlinks (need link building, not technical fixes).
